# AGENTS 默认契约

本文件是本仓库内所有开发 Agent 的默认工作契约。任何 Unity 后端、玩法规则、UI 接入、场景、测试或文档工作，都必须先遵循本文件，再结合具体任务规格执行。

## 项目背景

本项目是一个 Unity VR 射击训练游戏 DEMO，目标是围绕现代军事训练模拟器搭建可交互原型。当前项目使用 Unity 2022.3.62f3、XR Interaction Toolkit、OpenXR、UGUI 和 TextMeshPro。

核心训练模块包括：

- 100m 精度射校靶
- 移动目标射击
- 堑壕射击
- 城镇攻防
- 武器库 / 武器墙
- 设置界面

UI 方案已经通过 draw.io 整理为可编辑线框图，BDD 文档已经把每个界面的用户行为和验收场景落成中文规格，接口文档已经把 UI 前端层与 Unity 后端服务层的数据交互约束落成契约。最新仓库已将需求、BDD、接口和任务规格统一收拢到 `docs/`。

## 后端开发硬约束

所有 Unity 后端开发必须遵循 BDD 和接口文档。

- 不允许绕过 BDD 直接按个人理解实现玩法。
- 不允许 UI View 直接计算评级、胜负、敌人生成、弹药规则或训练结算。
- 不允许 HUD 显示写死示例数据；HUD 必须读取 DTO 或订阅事件。
- 不允许按钮直接创建敌人、切换 AI 或修改场景核心状态。
- 不允许随机逻辑没有固定随机种子入口。
- 不允许训练结算从 UI 文本反推数据。
- 不允许服务接口与 `docs/接口文档/` 定义不一致，除非同步更新文档。
- 不允许脱离 `docs/任务规格/` 私自扩大或缩小任务范围。
- 不允许把 Later/P2/P3 内容作为 P1 的隐性前置条件。

## 阶段与任务规格约束

产品按三阶段交付：

- P1：100m 射校原型闭环。
- P2：移动靶与全局系统扩展。
- P3：小型堑壕与城镇战斗原型。

任何实现任务必须先确认所属阶段，再确认负责人目录：

- `UI`
- `场景`
- `功能A-玩法流程与规则`
- `功能B-武器输入与AI基础`

任务执行规则：

1. 优先打开 `docs/任务规格/README.md`。
2. 找到对应阶段目录，例如 `docs/任务规格/阶段一-P1-100m射校原型闭环/`。
3. 找到对应负责人目录和具体 `task*.md`。
4. 按任务规格中的 BDD 引用、接口文档引用、交付物、测试要求和联调对象执行。
5. 任务编号在同一阶段内全局唯一，不得随意改名、复用或跳号。
6. 如果任务规格缺少必要 BDD 或接口引用，应先补文档再实现。

P1 当前优先级最高，必须先保证 100m 射校闭环可在 Unity Editor 无 VR 环境中完成测试。

## 推荐架构边界

遵循 `docs/接口文档/00-前后端交互总约束.md` 中的分层：

- UI View：只负责显示、按钮状态和视觉反馈。
- UI Presenter：把服务 DTO 渲染到 View，并转发 UI 事件。
- 应用服务：负责页面导航、Session、设置、武器和结算协调。
- 玩法服务：负责射击、弹着、移动靶、敌人、队友、搜索进度等规则。
- Unity 场景层：负责 Prefab、Animator、XR Interactor、NavMesh、碰撞体等 Unity 对象。

前端 UI 层与后端玩法层必须使用“命令 + 查询 + 事件”交互：

- 命令：UI 请求后端执行行为。
- 查询：UI 获取当前状态 DTO。
- 事件：后端通知 UI 刷新状态。

## 开发流程

每个开发任务必须按以下顺序推进：

1. 找到对应 `docs/任务规格/**/task*.md`，确认阶段、负责人、交付物和联调对象。
2. 找到相关 BDD 文件，确认要实现的行为场景。
3. 找到相关接口文档，确认 DTO、服务方法、事件和错误码。
4. 先补或更新测试，再实现业务逻辑。
5. 按接口文档实现服务和数据模型。
6. 通过 Presenter 把 DTO 接到 UI，不在 View 中写业务逻辑。
7. 运行 EditMode 或 PlayMode 测试。
8. 如果行为、接口或任务范围发生变化，同步更新 BDD、接口文档和任务规格。

## 测试契约

后端开发至少需要覆盖：

- EditMode Test：DTO、状态机、评级算法、设置持久化、武器和弹药规则。
- PlayMode Test：页面跳转、按钮点击、HUD 刷新、场景加载、训练闭环。
- VR 实机验收：UI 可读性、舒适度、射击手感、队友遮挡、HUD 视野。

每个拆分出来的功能完成时，必须同步交付可追溯到 BDD 场景和任务规格的测试：

- 纯业务规则、DTO、算法、状态机、服务命令和错误码必须提供 EditMode 单元测试。
- 涉及 UI、场景、XR 输入替身、HUD 刷新、页面跳转或训练闭环的功能必须提供 PlayMode 测试。
- 如果功能依赖真实 VR 设备，必须同时提供无 VR 设备可运行的输入替身测试路径，并在交付说明中列出 VR 实机验收项。
- 不允许把“功能已实现但测试后补”作为完成状态；测试缺失时，该功能只能标记为未完成或部分完成。
- 如果确实无法自动化测试，必须在对应 BDD 或交付说明中写明原因、手工验收步骤和后续补测计划。

所有测试场景应能追溯到 `docs/BDD/screens/*.feature.md` 中的具体场景。

无 VR 设备时，必须能通过 Unity Editor、XR Device Simulator 或输入测试替身完成核心流程测试。业务服务不得直接读取硬件输入，输入必须通过可替换输入适配层进入。

## Unity 命名和绑定

必须遵循 `docs/接口文档/11-Unity场景与Prefab约定.md`：

- 页面根节点：`Screen_<ScreenId>`
- 按钮：`Button_<ScreenId>_<Action>`
- 文本：`Text_<ScreenId>_<Data>`
- HUD 元素：`Hud_<HudType>_<Name>`
- 素材占位：`Placeholder_<ScreenId>_<Asset>`

关键 UI 控件和文本必须提供稳定测试 ID，方便 PlayMode Test 自动定位。

## 开发交付标准

一个后端任务完成时，应至少满足：

- 对应任务规格中的交付物已完成。
- 对应 BDD 场景已实现或有明确未实现说明。
- 对应接口文档中的 DTO、服务方法、事件和错误码已实现。
- UI 不包含后端业务规则。
- 训练数据、HUD 数据和结算数据都来自服务层。
- 每个拆分功能的相关 EditMode/PlayMode 测试已添加或更新，并能追溯到对应 BDD 场景。
- 与任务规格中列出的联调对象完成接口或测试替身对齐。
- 文档和代码没有明显冲突。

## 最小交付优先级

后端团队优先实现 P1 闭环所需内容：

1. 通用 DTO、错误码、事件总线。
2. 页面导航与 UI 事件。
3. 训练 Session 数据模型。
4. HUD 显示数据聚合。
5. 输入适配、基础武器、弹药服务。
6. 100m 射校服务、HUD、弹着分析和最终评级。
7. P1 对应 UI、场景和 PlayMode/EditMode 测试。

P2 再推进移动目标、武器库原型、设置原型和 HUD/弹药复用。P3 再推进小型堑壕、小型城镇、简单敌人 AI、3 人纵队和搜索结算。

详细拆分见 `docs/接口文档/12-后端开发交付清单.md`。
